## 1. The Heatmap

### What It Is

A colored grid where **rows are feature categories** (Cloud, 2D Modeling, Rainfall, etc.) and **columns are software versions** (v1.5 through v2027.0). Each cell's color intensity represents how many features shipped in that category for that version. Darker = more features. Empty = nothing shipped.

### What Question It Answers

**"Where has development effort been concentrated, and how has that changed over time?"**

This is a question nobody can answer by reading a feature list. You'd have to mentally classify 809 features into categories, count them per version, and hold the pattern in your head. The heatmap does that instantly.

### What It Looks Like With Real ICM Data

                    2011────────2015────────2019────────2023─────2025

### The Five Stories This Heatmap Tells

**Story 1 — The Cloud Pivot:** The ☁️ row is completely empty for 12 years (2011–2022), then erupts with maximum intensity at v2024.0 and sustains through 2025. Cloud went from nonexistent to the single largest development investment in a single release cycle. No other category in ICM's history shows this pattern.

**Story 2 — The 1D→2D Shift:** In early versions, the 🔧 1D Hydraulics row is consistently among the densest. By 2020–2021, 🌊 2D Modeling has overtaken it. By 2024, 1D is nearly empty. The field migrated from pipe network analysis to surface flood modeling.

**Story 3 — Rainfall Matured and Stopped:** The 🌧️ row shows steady activity from 2017–2021 (Euler Type II, FEH2013, NOAA ATLAS 14), then goes completely dark. The rainfall generation toolkit reached completeness and investment stopped.

**Story 4 — SWMM Was a Burst:** The 💧 row shows a single intense spike at v10.5 (2019), then scattered minor touches. SWMM arrived as a complete system in one release — a fundamentally different development pattern from everything else.

**Story 5 — Water Quality Peaked Early:** The 🧪 row was densest in 2015–2016 (sediment modeling, 1D diffusion), then tapered to nothing. Either the capability matured or market demand didn't justify continued investment.

### The Controls That Transform Understanding

**Normalize by row** changes what the colors mean. Without normalization, Cloud's massive counts dominate the palette — everything else looks pale. With normalization, each row scales to its own maximum, so you can see patterns *within* every category, not just the biggest ones. Suddenly the Rainfall row shows its own internal story of growth and plateau.

**Color scheme** switches between category-colored cells (each row in its own hue), single-scale blue (for comparing across categories), heat gradient (yellow→red, intuitive for "hotness"), or viridis (perceptually uniform for accessibility).

**Show values** toggles the numeric counts inside each cell. With values shown, you can read exact numbers. Without them, you rely purely on color pattern recognition — better for spotting macro trends.

## 2. The Chains

### What It Is

A chain is a sequence of related features that appear across multiple versions, connected because they represent the **same capability evolving over time**. The chain detector scans all 809 features and identifies these threads automatically by analyzing:

- **Explicit links** between features that reference each other
- **Title similarity** (Jaccard coefficient > 0.55) between features in different versions
- **Shared rare keywords** that appear in 2–8 features across multiple versions

### What Question It Answers

**"How did this capability actually develop? What's the story behind these scattered feature entries?"**

A feature list shows 14 entries called "Cloud Capabilities" across 14 versions. That tells you nothing about the *arc* — when it was introduced, how it grew, whether it's still active, and what changed at each step. A chain turns those 14 isolated entries into a single narrative.

```
🌰 Introduced    — Chain has only 1 feature (just appeared)

This answers: "Should I expect this capability to keep improving, or is it done?"

## 3. The Comparison

### What It Is

A tool that lets you pick two versions and see **everything that changed between them** — organized by timeline, by category, or by evolution chains. It produces a structured diff of the product's development between any two points in its 14-year history.

### What Question It Answers

**"If I upgrade from version X to version Y, what exactly do I get?"**

This is the most common question ICM users ask, and today it requires reading through every intermediate release note manually. The comparison compresses that into a single structured view.

## 4. How All Three Interconnect

### The Core Principle

Each feature answers a different *type* of question, and the answers to one type naturally lead to questions of another type:

```
HEATMAP answers: "WHERE is the pattern?"
   ↓ leads to
CHAINS answer: "WHY does this pattern exist?"
   ↓ leads to  
COMPARISON answers: "WHAT does this pattern mean FOR ME?"
```

### The Six Connections

**Connection 1: Heatmap → Chains (Pattern → Explanation)**

```
User SEES in heatmap:  ☁️ Cloud row is empty for 12 years,
                       then explodes in 2023

User ASKS:             "Why? What happened?"

Chains SHOW:           Cloud Platform chain (14 versions)
                       with the full arc from introduction
                       through growth to maturation

The heatmap reveals the PATTERN.
The chain explains the NARRATIVE behind it.
```

Technical connection: Clicking a heatmap cell or row header dispatches an event that filters the chain explorer to show only chains in that category, expanded to show the relevant version node.

**Connection 2: Chains → Heatmap (Narrative → Context)**

```
User READS chain:      Rainfall chain ended at v2021.7
                       with status 🏔️ Stable

User ASKS:             "What replaced it? Where did
                       development effort go instead?"

Heatmap SHOWS:         The ☁️ Cloud row ignited at v2024.0 —
                       exactly when 🌧️ Rainfall went dark.
                       Resources shifted from rainfall to cloud.

The chain reveals WHEN something stopped.
The heatmap reveals WHAT took its place.
```

Technical connection: Clicking a chain node highlights the corresponding cell in the heatmap, and the chain's version range overlays a highlight band on the heatmap's columns so you can see what else was happening in the same period.

**Connection 3: Heatmap → Comparison (Pattern → Decision)**

```
User SEES in heatmap:  Massive development activity between
                       v2024.0 and v2026.3

User ASKS:             "If I upgrade to capture that activity,
                        what exactly do I get?"

Comparison SHOWS:      380 features, 5 major releases,
                       Cloud as #1 category, 1D declining

The heatmap identifies WHERE the value is.
The comparison quantifies WHAT that value contains.
```

Technical connection: Clicking a heatmap column header pre-fills the comparison's "from" or "to" version dropdown. The heatmap essentially becomes a visual version-picker — you click the column where you see interesting activity, and the comparison tool opens with that version selected.

**Connection 4: Comparison → Chains (Diff → Narrative)**

```
User RUNS comparison:  v10.5 → v2026.3 shows ~380 features

User ASKS:             "Are these 380 independent features,
                       or are they connected in some way?"

Chains SHOW:           4 major chains span this range:
                       Cloud (14 nodes), 2D (4 nodes),
                       Rainfall (3 nodes), SWMM (2 nodes)

                       So ~180 of those 380 features are
                       actually just 4 evolving capabilities,
                       not 180 separate things.

The comparison counts WHAT changed.
The chains show HOW those changes relate to each other.
```

Technical connection: The comparison engine runs chain detection on the features within the compared range and displays them in the "By Chains" view mode. It also annotates the unified timeline view with chain membership badges so users can see which features belong to which arc.

**Connection 5: Chains → Comparison (Narrative → Scope)**

```
User READS chain:      Subgrid/2D Mesh spans v1.5 → v2027.0

User ASKS:             "If I want to get the full subgrid
                       capability, which versions do I need?"

Comparison SHOWS:      Pre-filled with v1.5 → v2027.0,
                       filtered to 2D Modeling category:
                       the exact set of features that 
                       constitute the subgrid capability

The chain defines the SCOPE of a capability.
The comparison lists every feature within that scope.
```

Technical connection: Each chain card has a "Compare this range" button that opens the comparison tool pre-filled with the chain's from/to versions and filtered to the chain's primary category.

**Connection 6: Comparison → Heatmap (Decision → Context)**

```
User RUNS comparison:  v10.5 → v2026.3
                       sees Cloud ↑32%, 1D ↓14%

User ASKS:             "Is the 1D decline a trend that will
                       continue, or just a dip?"

Heatmap SHOWS:         The entire 🔧 1D row across all 48
                       versions — revealing that 1D was densest
                       in 2011-2016, has been declining since,
                       and shows no sign of reversal.
                       The decline is structural, not cyclical.

The comparison detects A SHIFT.
The heatmap reveals whether it's TEMPORARY or PERMANENT.
```

Technical connection: Running a comparison overlays a highlight band on the heatmap marking the compared range. Cells inside the range are full-brightness; cells outside are dimmed. The category shift indicators in the comparison banner link directly to the corresponding heatmap rows.

```

The event bus is the nervous system. Each feature operates independently on the shared data, but when a user acts on any one of them, events propagate to the others so they update their state to stay in sync. The user never has to manually coordinate — clicking in the heatmap automatically prepares the chains and comparison to answer the next natural question.

### Why This Trio Works

The three features form a **cognitive loop** that matches how people actually think about complex product histories:

| Cognitive Need | Feature | Output |
|---|---|---|
| **Orientation** — "What's the landscape?" | Heatmap | Macro patterns across all 48 versions |
| **Understanding** — "What does this mean?" | Chains | Narrative arcs explaining why patterns exist |
| **Decision** — "What should I do?" | Comparison | Structured diff with export for action |

No single feature can serve all three needs. A heatmap without chains is a pattern without explanation. Chains without a heatmap lack context. A comparison without either is a list without meaning. Together, they transform 809 isolated feature entries into a navigable, understandable, actionable history of the product.
